## Universal Commands

```bash
pnpm install                        # after pulling or changing deps
pnpm infra                          # start Postgres + Redis via Docker
pnpm start                          # start everything (infra → migrator → scheduler → api + web)
pnpm turbo-start                    # dev mode via Turborepo (assumes infra running)

pnpm --filter api tsc --noEmit      # typecheck API
pnpm --filter web tsc --noEmit      # typecheck web

# After API endpoint/schema changes:
pnpm --filter api build:types       # rebuild API declarations (REQUIRED)
```

## Never Do

- Use npm or yarn. pnpm ≥9.5.0 only.
- Use Node < 22.13.0. See `.nvmrc`.
- Use `src/...` prefix in API imports — `tsconfig.baseUrl` is `src`, use `@/` alias.
- Add env vars without updating the Zod config schema.
- Write inline string enums in Zod schemas — use constants from `app-constants`.
- Manually edit `router.ts`, `contract.ts`, `resource-client.ts` — they are auto-generated.

## Type Flow: API → Web

1. API endpoints define `.input(zodSchema).output(zodSchema).handler(...)`.
2. `pnpm --filter api build:types` builds `.d.ts` files to `apps/api/dist/`.
3. Web imports types via `"api": "workspace:*"` dependency — `import type { AppClient } from 'api'`.
4. No codegen, no shared types package. Types flow through TypeScript declarations.

## Filesystem-Based Routing (API)

Endpoints live in `resources/<name>/endpoints/`. The router is auto-generated by `scripts/codegen-router.ts`:

```bash
cd apps/api && npx tsx scripts/codegen-router.ts    # regenerate router
```

Conventions:
- `list.ts` → GET, `create.ts` → POST, `get.ts` → GET, `update.ts` → PUT, `delete.ts` → DELETE
- `name.post.ts` → POST `/name`, `name.get.ts` → GET `/name`
- `[param]/` → `{param}` dynamic segment
- Non-param subdirs inside `endpoints/` create nested router groups (camelCased)

Example: `users/endpoints/[userId]/get.ts` → `orpcClient.users.get({ userId: '...' })`
